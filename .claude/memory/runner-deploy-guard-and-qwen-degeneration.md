**SUBMODULE FOOTGUN — FIXED 2026-06-06 (`make vendor-sync` + pin + pre-commit hook):** the recurring "deploy blocked because a `git commit -a` from a drifted local checkout reverted `vendors/llama.cpp` to an old SHA" is now solved. Source of truth is `LLAMA_CPP_SHA` in the Makefile (= the rolled-forward, gemma4uv-capable SHA). `make vendor-sync` now FORCES the submodule to that pin + stages the gitlink + runs the gemma4uv check (it used to just `git submodule update` = sync TO the already-wrong gitlink, which is why it "didn't work"). `make vendor-check` asserts staged gitlink == pin. `make install-hooks` (run once per clone — core.hooksPath isn't committed) installs a pre-commit hook that BLOCKS committing a drifted gitlink. To roll llama.cpp forward: bump `LLAMA_CPP_SHA`, then `make vendor-sync && git commit`. deploy.yml now has `paths-ignore` (Makefile/scripts-hooks/.github/**.md) so tooling commits don't roll the runner.

**llmmllab-runner deploy guard (`.github/workflows/deploy.yml`):** the "Force submodules to the committed SHA" step is NOT a SHA-equality check — after `git submodule update --init --force --recursive` it greps `vendors/llama.cpp/tools/mtmd` for the literal string **`gemma4uv`** and FAILS the deploy if absent. It's a capability gate proving the llama.cpp submodule wasn't reverted to a stale/older build (the gemma-4 vision model file was renamed `gemma4v.cpp`→`gemma4uv.cpp` in early-June-2026 llama.cpp, so any SHA before that rename fails the check). This is LOAD-BEARING — do NOT remove it. On 2026-06-06 a `git commit -a` in a stale local checkout silently rolled `vendors/llama.cpp` back 12 days (308f61c31 → 549b9d84); the guard correctly blocked that deploy. Fix when blocked: restore the intended gitlink (`cd vendors/llama.cpp && git checkout <rolled-forward-sha> && cd ../.. && git add vendors/llama.cpp && git commit`). The known-good rolled-forward SHA as of 2026-06-06 is `308f61c31` (has gemma4uv). Backend is CUDA (`-DGGML_CUDA=ON`). `.models.yaml` is baked into the image; an image rebuild keyed by `LLAMA_SHA` only recompiles llama.cpp when the submodule SHA changes, so a `.models.yaml`-only change redeploys fast.

**Qwen3.6-27B degeneration modes (it's a hybrid-SSM arch):**
- Earlier (fixed ~2026-06-05): word-association runaway / repeated tool-call fences on the Q6 quant — addressed by making the UD Q5_K_XL the canonical `Qwen3_6_27B`, wiring samplers through llama.cpp, num_predict cap, and ADDING a DRY penalty.
- New (2026-06-06, under test): progressive CHARACTER-SCRAMBLING of repeated tokens (file paths, repo names, even `2>&1`→`2>&9`), worsening as context fills, in LONG interactive sessions. Short automated cron dispatches (engineer/reviewer PRs) stay CLEAN → the trigger is context accumulation, not a constant defect. Prime suspect: the DRY penalty (`dry_allowed_length: 2`, `dry_penalty_last_n: 2048`) penalizing repeated identifiers. Test deployed: `dry_multiplier 1.2→0` + `think: False` on the canonical block. Backup theory: llama.cpp post-2026-06-03 SSM regression (ggml-org/llama.cpp#24168, reported on SYCL; we're CUDA). #24201 (Hexagon) is NOT us.
- Canonical sampler (pre-test): temp 0.60, top_k 12, top_p 0.95, min_p 0.06, num_predict 16384, dry_multiplier 1.2/base 1.75/allowed_length 2/penalty_last_n 2048. If DRY is the culprit, also disable on `_MTP`/`_Q6` and revisit the word-cascade fix with gentler DRY.
